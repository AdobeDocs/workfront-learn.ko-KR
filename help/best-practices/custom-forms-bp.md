# 모범 사례 - 사용자 정의 양식

## Adobe Workfront “모범 사례”란 무엇입니까?

모범 사례는 효과적이고 효율적인 작업 과정을 나타내는 지침입니다. 사용자와 귀사의 사용자가 쉽게 채택할 수 있으며 조직 전체에서 복제할 수 있습니다.

이러한 권장 사항을 검토할 때 일부 Workfront 모범 사례는 보편적인 반면 다른 모범 사례는 주제에 더 구체적일 수 있음을 명심하십시오. Workfront 시스템을 올바르게 설정하고 사용하는 데 있어 이들 모범 사례를 프레임워크로 사용하십시오.

## 이 페이지 탐색

이 페이지를 스크롤하면 먼저 해당 주제에 대한 모든 모범 사례의 상위 수준 목록을 찾을 수 있습니다. 이를 통해 “이유”에 대한 세부 사항을 자세히 살펴보지 않고도 권장 사항을 검토할 수 있습니다.

“이러한 모범 사례를 채택하는 이유는 무엇입니까?” 영역(상위 수준의 목록 뒤)에서는 일부 모범 사례와 프로세스 및 도구 등으로 간주되는 이유에 대한 자세한 설명이 나와 있으며, Workfront 인스턴스를 사용하여 구현하는 것을 고려해야 합니다.

</br>
</br>

## 사용자 정의 양식 모범 사례

* 사용자 정의 양식을 먼저 화이트보드나 종이에 도표화한 다음 Adobe Workfront에서 만듭니다.

* 프로젝트 템플릿으로 만든 프로젝트에 항상 특정 사용자 정의 양식을 첨부해야 하는 경우 해당 템플릿에 사용자 정의 양식을 첨부합니다. 거의 변경되지 않거나 특정 정보가 필요한 필드를 작성할 수도 있습니다.

* Workfront 시스템에 회사로 추가한 외부 고객 또는 공급업체에 대한 정보를 추적하는 사용자 정의 양식을 만듭니다.

* 개별 필드를 여러 양식에 추가하는 대신 공통 필드와 계산된 필드를 포함하는 단일 “일반” 또는 “범용” 사용자 정의 양식을 만듭니다. 이를 통해 일관된 방식으로 전사적 정보를 수집할 수 있습니다.

* 사용자 정의 양식에 섹션 구분을 추가하여 체계적이고 이해하기 쉽게 유지합니다.

* 양식을 완전히 작성하고 필요한 모든 정보를 얻을 수 있도록 사용자 정의 양식을 짧게 유지합니다.

* 라디오 버튼 및 드롭다운 메뉴와 같이 미리 정의된 사용자 정의 필드 옵션을 사용하여 사용자가 특정 옵션 중에서 선택하도록 함으로써 응답 옵션을 제한합니다.

* 필드 레이블이 명확하고 설명적인지 확인합니다.

* 사용자 정의 필드의 지침 필드에 정보를 추가하여 필드를 작성하는 사람이 입력해야 하는 정보를 나타냅니다.

* 다른 필드가 특정 방식으로 작성될 때 필요한 필드를 표시하려면 표시 논리를 사용합니다. 사용자 정의 양식에서 건너뛰기 논리를 사용하여 요청 유형과 관련 없는 필드를 숨깁니다.

* 다른 양식에서 계산된 정보를 가져와 사용자 정의 양식에 필요한 계산 수를 줄입니다.

* 사용자 정의 양식에 대한 새 필드를 만들기 전에 필드 라이브러리를 확인하여 해당 필드가 이미 있는지 확인합니다.

* 필수 필드를 사용하여 프로세스 또는 조직에 중요한 정보를 캡처합니다.

* 사용자 정의 양식에서 필드 이름을 변경할 때는 해당 필드를 호출하는 계산된 필드에 영향을 미칠 수 있으므로 주의합니다.

* 정기적인 Workfront 시스템 유지 관리의 일환으로 사용자 정의 양식 및 필드를 정기적으로(예: 분기에 한 번) 검토합니다.

* 불필요한 사용자 정의 양식은 삭제하는 대신 비활성화합니다.

* 섹션 구분 아래의 사용자 정의 양식에서 더 이상 필요하지 않은 사용자 정의 필드를 숨깁니다. 그런 다음 해당 섹션을 시스템 관리자만 볼 수 있도록 합니다.

* Workfront 인스턴스에서 사용자 정의 양식을 만들 수 있는 사람을 제한합니다.

## 이러한 모범 사례를 채택하는 이유는 무엇입니까?

**모범 사례**

사용자 정의 양식을 먼저 화이트보드나 종이에 도표화한 다음 Adobe Workfront에서 만듭니다.

**이유**

모든 필수 정보에 대한 필드가 있고, 사용자가 양식을 쉽게 작성할 수 있는 방식으로 필드가 구성되어 있는지 확인합니다. 또한 양식을 도표화하면 표시 논리를 사용하여 일부 필드를 숨기거나 표시할 수 있는지 여부를 결정하는 데 도움이 될 수 있습니다.

</br>
</br>

**모범 사례**

프로젝트 템플릿으로 만든 프로젝트에 항상 특정 사용자 정의 양식을 첨부해야 하는 경우 해당 템플릿에 사용자 정의 양식을 첨부합니다. 거의 변경되지 않거나 특정 정보가 필요한 필드를 작성할 수도 있습니다.

**이유**

이렇게 하면 양식이 이미 첨부되고 일부 정보가 이미 작성되어 있으므로, 프로젝트를 만드는 속도가 빨라지고 해당 사용자 정의 필드가 모든 관련 프로젝트에서 정확하고 완전하게 작성됩니다.

</br>
</br>

**모범 사례**

Workfront 시스템에 회사로 추가한 외부 고객 또는 공급업체에 대한 정보를 추적하는 사용자 정의 양식을 만듭니다.

**이유**

사용자 정의 양식을 사용하여 주소, 기본 연락처 이름 등을 추적하면 Workfront 내에서 쉽게 액세스할 수 있습니다. 또한 이 사용자 정의 양식 정보를 보고서로 가져올 수 있습니다.

</br>
</br>

**모범 사례**

개별 필드를 여러 양식에 추가하는 대신 공통 필드와 계산된 필드를 포함하는 단일 “일반” 또는 “범용” 사용자 정의 양식을 만듭니다. 이를 통해 일관된 방식으로 전사적 정보를 수집할 수 있습니다.

**이유**

“일반” 양식을 사용하면 이러한 필드가 모두 한 곳에 있기 때문에 유지 관리가 더 쉬워집니다. 단일 양식을 업데이트하기만 하면 되며, 이러한 개별 필드를 여러 다른 양식에 저장한 다음 개별적으로 업데이트할 필요가 없습니다.

</br>
</br>

**모범 사례**

사용자 정의 양식에 섹션 구분을 추가하여 체계적이고 이해하기 쉽게 유지합니다.

**이유**

관련 정보를 섹션으로 그룹화하면 사용자가 양식을 탐색하는 데 도움이 됩니다.

</br>
</br>

**모범 사례**

양식을 완전히 작성하고 필요한 모든 정보를 얻을 수 있도록 사용자 정의 양식을 짧게 유지합니다.

**이유**

긴 양식은 사용자에게 부담을 줄 수 있으며, 이 때문에 양식이 완전히 작성되지 않는 경우가 많습니다. 결과적으로 할당 정보가 불완전해지고 보고할 데이터가 부정확해집니다.

사용자 정의 양식에 많은 필드가 있는 경우 가능하면 관련 필드를 나란히 배치하여 사용자가 덜 스크롤하도록 합니다. 건너뛰기 논리를 사용하여 작성할 필요가 없는 필드를 숨기거나 표시 논리를 사용하여 특정 필드를 표시할 수도 있습니다.

</br>
</br>

**모범 사례**

라디오 버튼 및 드롭다운 메뉴와 같이 미리 정의된 사용자 정의 필드 옵션을 사용하여 사용자가 특정 옵션 중에서 선택하도록 함으로써 응답 옵션을 제한합니다.

**이유**

미리 정의된 필드를 사용하면 사용자가 상자를 클릭하거나 메뉴에서 선택하면 되기 때문에 해당 질문에 대한 모든 답변이 동일해집니다.

일관되고 정확한 데이터는 정확한 보고에 필수적입니다. 데이터가 일관되지 않으면 보고서가 부정확해져 개인 차원부터 그 위쪽까지 의사결정에 영향을 미칠 수 있습니다. 또한 이렇게 일관된 데이터를 사용하면 보고서에 차트를 추가하여 데이터를 시각적으로 표현할 수 있습니다. 개방형 텍스트 필드는 차트에서 사용할 수 없습니다.

</br>
</br>

**모범 사례**

필드 레이블이 명확하고 설명적인지 확인합니다.

**이유**

이렇게 하면 사용자 정의 양식을 작성하는 사람이 자신에게 요청된 정보를 이해할 수 있습니다.

</br>
</br>

**모범 사례**

사용자 정의 필드의 지침 필드에 정보를 추가하여 필드를 작성하는 사람이 입력해야 하는 정보를 나타냅니다.

**이유**

이 정보는 ? 위에 마우스를 올려 놓으면 팝업으로 나타납니다. 해당 아이콘은 사용자 정의 양식의 필드 옆에 있습니다. 텍스트 필드에 필요한 모든 형식을 포함하여 필드에 입력해야 하는 데이터를 포함합니다.

사용자에게 더 자세한 정보를 제공하면 추가 대화, 주고받는 이메일 또는 혼란을 줄이는 데 도움이 됩니다. 정보가 불완전하거나 누락되면 작업이 지연될 수 있습니다.

</br>
</br>

**모범 사례**

다른 필드가 특정 방식으로 작성될 때 필요한 필드를 표시하려면 표시 논리를 사용합니다. 사용자 정의 양식에서 건너뛰기 논리를 사용하여 요청 유형과 관련 없는 필드를 숨깁니다.

**이유**

필요한 필드만 표시하거나 필요하지 않은 필드를 숨기면 사용자 정의 양식이 더 깔끔해지고, 사용자가 사용자 정의 양식을 작성할 때 혼동이 줄어듭니다. 또한 전체적으로 양식이 짧아져 사용자에게 부담이 덜 되며 더 높은 응답률을 촉진합니다.

표시 논리를 사용하면 만들고 유지 관리해야 하는 사용자 정의 양식의 수를 줄이는 데도 도움이 됩니다.

</br>
</br>

**모범 사례**

다른 양식에서 계산된 정보를 가져와 사용자 정의 양식에 필요한 계산 수를 줄입니다.

**이유**

예를 들어 항목에 연결된 자산 수를 계산하는 문제 양식에 “자산 수”라는 계산된 필드가 있습니다. 이 필드는 요청 대기열에서 사용됩니다. 요청이 변환될 때 프로젝트로 이월하려면 이 정보가 필요합니다. 문제 양식을 복사하여 프로젝트 양식으로 저장합니다. 그런 다음 문제 계산된 필드의 이름을 프로젝트 양식의 계산 상자에 추가합니다. 이 예에서는 “자산 수”라는 프로젝트 양식의 계산된 필드에 문자 그대로 “자산 수”를 입력합니다. 이렇게 하면 Workfront가 프로젝트에서 이 계산을 다시 실행하지 않고 대신 문제 사용자 정의 양식의 값을 사용합니다.

</br>
</br>

**모범 사례**

해당하는 경우 동일한 용도로 사용되는 필드 라이브러리의 기존 필드를 사용합니다.

**이유**

Workfront에서는 두 필드의 이름이 동일할 수 없습니다. 필드에 공통 이름이 있다면 해당 필드가 이미 존재할 가능성이 있습니다. 새 필드를 만들기 전에 필드 라이브러리를 확인하여 필드가 이미 존재하는지 확인하십시오.

본인이 직접 만들지 않은 필드를 사용하는 경우 이 필드를 수정하면 해당 필드가 속한 모든 사용자 정의 양식에 영향을 미친다는 점을 기억하십시오. 계산을 변경하거나 필드 유형을 변경해야 하는 경우(예: 텍스트를 라디오 버튼으로) 새 필드를 만들고 원래 이름과 다른 이름을 지정해야 합니다. (이름이 유사한 필드가 여러 개 있으면 사용자가 혼동할 수 있습니다.)

</br>
</br>

**모범 사례**

필수 필드를 사용하여 프로세스 또는 조직에 중요한 정보를 캡처합니다.

**이유**

사용자 정의 양식의 데이터가 불완전하면 작업이 지연되고 보고에 영향이 미칠 수 있습니다. 필수 필드 표시기(필드 이름 옆의 빨간색 *)는 사용자 정의 양식을 편집한 다음 저장하거나 공식적으로 요청을 제출하기 전에 특정 정보가 필수임을 사용자에게 상기시킵니다.

그러나 필수 필드는 꼭 필요할 때에 한해 신중하게 사용해야 합니다. 모든 필드를 필수로 지정하면 사용자가 유용하고 완전한 정보로 필드를 작성하는 데 장애물이 될 수 있습니다. 또한 오브젝트의 세부 정보 영역에서 사용자 정의 양식을 편집할 때 필수 필드가 불완전하면 사용자 정의 양식을 저장할 수 없습니다.

</br>
</br>

**모범 사례**

사용자 정의 양식에서 필드 이름을 변경할 때는 해당 필드를 호출하는 계산된 필드에 영향을 미칠 수 있으므로 주의합니다.

**이유**

필드의 이름을 변경할 때 사용자 정의 양식의 계산된 사용자 정의 필드 또는 텍스트 모드에 내장된 계산에서 사용되는 이름을 업데이트해야 합니다. 필드 이름을 변경하면 계산이 중단되고 부정확한 정보가 생성될 수 있습니다.

</br>
</br>

**모범 사례**

정기적인 Workfront 시스템 유지 관리의 일환으로 사용자 정의 양식 및 필드를 정기적으로(예: 분기에 한 번) 검토합니다.

**이유**

사용자 정의 양식 및 필드는 조직에서 작업을 수행하는 데 필요한 데이터를 수집하지 않는 경우 유용하지 않습니다.

업데이트할 때 이러한 변경 사항이 Workfront의 다른 측면에 어떤 영향을 미치는지 염두에 두십시오. 예를 들어 필드 이름을 변경하면 해당 필드를 사용하는 계산이 중단될 수 있습니다. 또는 양식의 레이블이나 이름을 변경하면 보고서에 필요한 정보가 표시되지 않거나 다른 시스템과의 통합이 실행되지 않을 수 있습니다.

사용자 정의 양식을 업데이트하는 것 외에도 사용량이 적거나 전혀 사용되지 않는 양식을 파악하십시오. 자주 사용하지 않는 양식을 다른 양식과 결합할 수 있을까요? 또는 양식에서 수집하는 정보가 팀에 더 이상 필요하지 않으므로 해당 양식의 사용을 중단해야 할 수 있습니다.

이러한 불필요한 양식에 대해 조치를 취하기 전에 사용자 정의 양식 및 사용자 정의 필드 비활성화와 삭제에 대해 읽어 보십시오.

</br>
</br>

**모범 사례**

불필요한 사용자 정의 양식은 삭제하는 대신 비활성화합니다.

**이유**

사용자 정의 양식을 삭제하면 해당 사용자 정의 양식을 통해 입력된 모든 사용자 정의 데이터가 삭제됩니다.
사용자 정의 양식을 비활성화하면 관련된 모든 내역 데이터가 유지됩니다. 즉, 이 정보에 대해 계속 보고할 수 있습니다.

또한 비활성화하면 사용자가 사용자 정의 양식을 선택하는 드롭다운 메뉴에 해당 양식이 표시되지 않습니다. 그러나 이미 첨부된 대상 오브젝트에는 양식이 여전히 표시됩니다.

</br>
</br>

**모범 사례**

섹션 구분 아래의 사용자 정의 양식에서 더 이상 필요하지 않은 사용자 정의 필드를 숨깁니다. 그런 다음 해당 섹션을 시스템 관리자만 볼 수 있도록 합니다.

**이유**

기존 사용자 정의 필드를 삭제하면 사용자 정의 양식에서 제거될 뿐만 아니라 해당 사용자 정의 필드가 사용되는 모든 위치에서 해당 필드에 포함된 모든 데이터가 삭제됩니다. 즉, 보고서에 필요할 수 있는 내역 데이터가 삭제됩니다.

데이터 손실을 방지하려면 사용자 정의 양식 자체에서 불필요한 필드를 숨기십시오. 사용자 정의 양식에서 섹션 구분을 사용하면 사용자가 연결된 Workfront 오브젝트에 대한 보기, 기여 또는 관리 액세스 권한이 있는지 여부에 따라 해당 섹션에 속한 필드를 숨기거나 노출할 수 있습니다. 또는 섹션을 ‘관리자만’으로 설정하여 시스템 관리자 액세스 권한 수준을 보유한 사용자만 양식의 전체 섹션을 보도록 할 수 있습니다.

</br>
</br>

**모범 사례**

Workfront 인스턴스에서 사용자 정의 양식을 만들 수 있는 사람을 제한합니다.


**이유**

사용자 정의 양식을 만들 수 있는 선별된 사용자 그룹이 있으면 Workfront 인스턴스에서 만들어지는 사용자 정의 양식의 수를 제어하는 데 도움이 됩니다.

또한 다른 사람이 양식을 만들 수 있도록 허용하면 시스템 관리자의 작업이 줄어들고, 그룹에서 각자 정기적으로 사용하는 사용자 정의 양식의 업데이트를 제어할 수 있습니다.
