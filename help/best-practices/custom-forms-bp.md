# 우수 사례 - 사용자 지정 양식

## Adobe Workfront &quot;모범 사례&quot;란 무엇입니까?

모범 사례는 효과적이고 효율적인 작업 과정을 나타내는 지침입니다. 는 사용자와 회사의 사용자가 쉽게 채택할 수 있습니다. 또한 조직 전반에 걸쳐 성공적으로 복제할 수 있습니다.

이러한 권장 사항을 검토할 때 일부 Workfront 우수 사례는 일반적이지만, 다른 권장 사항은 주제에 더 구체적일 수 있음을 유의하십시오. 이러한 우수 사례를 프레임워크로 사용하여 Workfront 시스템 설정 및 사용을 안내할 수 있습니다.

## 이 페이지 탐색

이 페이지를 스크롤하면 먼저 주제에 대한 모든 우수 사례 목록이 표시됩니다. 이렇게 하면 &quot;이유&quot;의 세부 사항으로 이동하지 않고 권장 사항을 검토할 수 있습니다.

&quot;이러한 우수 사례는 무엇입니까?&quot;가 상위 수준 목록 다음에 있는 영역은 일부 우수 사례에 더 자세한 정보를 제공하고 프로세스, 도구 등으로 간주되는 이유를 Workfront 인스턴스로 구현하는 것이 좋습니다.

</br>

## 사용자 지정 양식 우수 사례

* Adobe Workfront에서 만들기 전에 화이트보드나 종이 조각으로 사용자 지정 양식을 다이어그램합니다.

* 해당 템플릿으로 만든 프로젝트에 항상 특정 사용자 지정 양식이 첨부된 경우 사용자 지정 양식을 프로젝트 템플릿에 첨부합니다. 거의 변경되지 않거나 특정 정보가 필요한 필드를 입력할 수도 있습니다.

* 회사로 Workfront 시스템에 추가한 외부 고객 또는 공급업체에 대한 정보를 추적하는 사용자 지정 양식을 만듭니다.

* 개별 필드를 여러 양식에 추적하지 않고 공통 필드 및 계산된 필드를 포함하는 단일 &quot;일반&quot; 또는 &quot;범용&quot; 사용자 지정 양식을 만듭니다. 따라서 전사적인 정보를 일관된 방식으로 수집할 수 있습니다.

* 섹션 나누기를 사용자 지정 양식에 추가하여 구성하고 쉽게 이해할 수 있습니다.

* 사용자 지정 양식을 짧게 유지하여 양식이 완전히 채워지고 필요한 모든 정보를 얻을 수 있습니다.

* 라디오 단추 및 드롭다운 메뉴와 같이 사전 정의된 사용자 지정 필드 옵션을 사용하여 사용자가 특정 옵션 중에서 선택하도록 하여 응답 옵션을 제한합니다.

* 필드 레이블이 명확하게 표현되고 설명적인지 확인하십시오.

* 사용자 정의 필드의 지침 필드에 정보를 추가하여 필드를 입력해야 하는 정보를 지정합니다.

* 다른 필드가 특정 방식으로 채워질 때 필요한 필드를 표시하려면 표시 논리를 사용하십시오. 요청 유형과 관련이 없는 필드를 숨기려면 사용자 지정 양식의 건너뛰기 로직을 사용합니다.

* 다른 양식에서 계산된 정보를 가져와서 사용자 지정 양식에 필요한 계산 횟수를 줄입니다.

* 사용자 지정 양식에 대한 새 필드를 만들기 전에 필드 라이브러리를 확인하여 해당 필드가 이미 있는지 확인하십시오.

* 필수 필드를 사용하여 프로세스 또는 조직에 중요한 정보를 캡처합니다.

* 사용자 지정 양식에서 필드 이름을 변경할 때는 해당 필드를 호출하는 계산된 필드에 영향을 줄 수 있으므로 주의하십시오.

* 정기적으로 Workfront 시스템 유지 관리의 일부로 분기별로 한 번 등 사용자 지정 양식 및 필드를 정기적으로 검토합니다.

* 필요하지 않은 사용자 지정 양식을 삭제하는 대신 비활성화합니다.

* 섹션 브레이크 아래의 사용자 지정 양식에서 더 이상 필요하지 않은 사용자 지정 필드를 숨깁니다. 그런 다음 시스템 관리자만 해당 섹션을 볼 수 있도록 합니다.

* Workfront 인스턴스에서 사용자 지정 양식을 만들 수 있는 사용자를 제한합니다.

## 이러한 우수 사례는 무엇입니까?

**우수 사례**

Adobe Workfront에서 만들기 전에 화이트보드나 종이 조각으로 사용자 지정 양식을 다이어그램합니다.

**이유가 여기에 있습니다**

필요한 모든 정보에 대한 필드가 있고 사용자가 양식을 쉽게 작성할 수 있도록 필드가 구성되어 있는지 확인하십시오. 양식의 다이어그램은 표시 논리를 사용하여 일부 필드를 숨기거나 표시할 수 있는지 확인하는 데 도움이 될 수도 있습니다.

</br>
</br>

**우수 사례**

해당 템플릿으로 만든 프로젝트에 항상 특정 사용자 지정 양식이 첨부된 경우 사용자 지정 양식을 프로젝트 템플릿에 첨부합니다. 거의 변경되지 않거나 특정 정보가 필요한 필드를 입력할 수도 있습니다.

**이유가 여기에 있습니다**

이 방법으로 이미 양식이 첨부되어 있고 일부 정보가 이미 작성되어 있으므로 프로젝트 생성 시간을 단축하고 해당 사용자 지정 필드가 적용 가능한 모든 프로젝트에서 올바르게 완전히 입력되도록 할 수 있습니다.

</br>
</br>

**우수 사례**

회사로 Workfront 시스템에 추가한 외부 고객 또는 공급업체에 대한 정보를 추적하는 사용자 지정 양식을 만듭니다.

**이유가 여기에 있습니다**

사용자 지정 양식을 사용하여 주소, 기본 연락처의 이름 등을 추적하므로 Workfront 내에서 쉽게 액세스할 수 있습니다. 이 사용자 지정 양식 정보를 보고서에 가져올 수도 있습니다.

</br>
</br>

**우수 사례**

개별 필드를 여러 양식에 추적하지 않고 공통 필드 및 계산된 필드를 포함하는 단일 &quot;일반&quot; 또는 &quot;범용&quot; 사용자 지정 양식을 만듭니다. 따라서 전사적인 정보를 일관된 방식으로 수집할 수 있습니다.

**이유가 여기에 있습니다**

&quot;일반&quot; 양식은 이러한 필드를 모두 한 곳에 저장하므로 쉽게 유지 관리할 수 있습니다. 이러한 개별 필드가 여러 다른 형식으로 되어 있지 않고 단일 양식을 업데이트할 수 있으므로 개별적으로 업데이트해야 합니다.

</br>
</br>

**우수 사례**

섹션 나누기를 사용자 지정 양식에 추가하여 구성하고 쉽게 이해할 수 있습니다.

**이유가 여기에 있습니다**

관련 정보를 섹션으로 그룹화하면 사용자가 양식을 탐색할 수 있습니다.

</br>
</br>

**우수 사례**

사용자 지정 양식을 짧게 유지하여 양식이 완전히 채워지고 필요한 모든 정보를 얻을 수 있습니다.

**이유가 여기에 있습니다**

긴 양식은 사용자에게 위협적일 수 있으며 종종 양식이 완전히 작성되지 않을 수 있습니다. 결과적으로 할당 정보가 불완전하고 보고용 데이터가 부정확해집니다.

사용자 지정 양식에 필드가 많은 경우 가능한 한 관련 필드를 나란히 배치하여 사용자에게 스크롤하는 것이 적습니다. 논리 건너뛰기 를 사용하여 필드를 채우지 않아도 되거나 특정 필드를 표시하는 논리를 표시할 수도 있습니다.

</br>
</br>

**우수 사례**

라디오 단추 및 드롭다운 메뉴와 같이 사전 정의된 사용자 지정 필드 옵션을 사용하여 사용자가 특정 옵션 중에서 선택하도록 하여 응답 옵션을 제한합니다.

**이유가 여기에 있습니다**

사전 정의된 필드를 사용하면 사용자가 상자를 클릭하거나 메뉴에서 선택만 하므로 해당 질문에 대한 모든 답변이 동일합니다.

정확한 보고를 위해 일관되고 정확한 데이터가 필요합니다. 일관되지 않은 데이터는 부정확한 보고서를 생성하여 개별 수준의 결정에 영향을 줄 수 있습니다. 또한 이 일관된 데이터를 사용하면 데이터를 시각적으로 표시하기 위해 보고서에 차트를 추가할 수 있습니다. 열려 있는 텍스트 필드는 차트에서 사용할 수 없습니다.

</br>
</br>

**우수 사례**

필드 레이블이 명확하게 표현되고 설명적인지 확인하십시오.

**이유가 여기에 있습니다**

사용자 지정 양식을 작성하는 사람은 이러한 방식으로 요청되는 정보를 파악합니다.

</br>
</br>

**우수 사례**

사용자 정의 필드의 지침 필드에 정보를 추가하여 필드를 입력해야 하는 정보를 지정합니다.

**이유가 여기에 있습니다**

이 정보는 를 마우스로 가리키면 팝업으로 표시됩니다. 사용자 지정 양식의 필드 옆에 있는 아이콘. 텍스트 필드에 필요한 서식을 포함하여, 필드에 입력해야 하는 데이터를 포함합니다.

사용자에 대해 더 자세한 정보를 제공하면 추가 대화, 전후 이메일 또는 혼동을 줄일 수 있습니다. 정보가 불완전하거나 누락된 경우 작업이 지연될 수 있습니다.

</br>
</br>

**우수 사례**

다른 필드가 특정 방식으로 채워질 때 필요한 필드를 표시하려면 표시 논리를 사용하십시오. 요청 유형과 관련이 없는 필드를 숨기려면 사용자 지정 양식의 건너뛰기 로직을 사용합니다.

**이유가 여기에 있습니다**

필요한 필드만 표시하거나 그렇지 않은 필드를 숨기면 사용자 지정 양식이 명확해지고 사용자 지정 양식을 작성하면서 혼동을 줄일 수 있습니다. 이는 전체 짧은 양식에 기여하므로 사용자에게 위협이 되지 않고 응답률도 높아집니다.

표시 논리를 사용하면 만들고 유지 관리하는 데 필요한 사용자 지정 양식 수를 줄이는 데에도 도움이 될 수 있습니다.

</br>
</br>

**우수 사례**

다른 양식에서 계산된 정보를 가져와서 사용자 지정 양식에 필요한 계산 횟수를 줄입니다.

**이유가 여기에 있습니다**

예를 들어 항목에 첨부된 자산 수를 계산하는 문제 양식에 &quot;자산 수&quot;라는 계산된 필드가 있습니다. 요청 큐에서 사용됩니다. 요청이 변환될 때 이 정보를 프로젝트에 전달해야 합니다. 문제 양식의 사본을 만들고 프로젝트 양식으로 저장합니다. 그런 다음 문제 계산 필드의 이름을 프로젝트 양식의 계산 상자에 추가합니다. 이 예에서는 프로젝트 양식의 계산된 필드 &quot;자산 수&quot;에서 문자 그대로 &quot;자산 수&quot;를 입력합니다. 따라서 Workfront에서 프로젝트에서 이 계산을 다시 실행하려고 하지 않고 대신 문제 사용자 지정 양식의 값을 사용합니다.

</br>
</br>

**우수 사례**

해당되는 경우 같은 목적을 제공하는 필드 라이브러리의 기존 필드를 사용합니다.

**이유가 여기에 있습니다**

Workfront의 두 필드에는 동일한 이름을 사용할 수 없습니다. 필드에 공통 이름이 있는 경우 필드가 이미 있을 수 있습니다. 새 필드를 만들기 전에 필드 라이브러리를 확인하여 필드가 이미 있는지 확인하십시오.

만들지 않은 필드를 사용하는 경우 이 필드의 수정 사항이 ALL 사용자 지정 양식에 영향을 주므로 필드의 일부가 됩니다. 계산을 변경하거나 필드 유형(예: 텍스트를 라디오 단추)을 변경해야 하는 경우, 새 필드를 만들어 원래 이름과 다른 이름을 지정해야 합니다. (이름이 비슷한 여러 필드가 사용자에게 혼동을 줄 수 있음을 명심하십시오.)

</br>
</br>

**우수 사례**

필수 필드를 사용하여 프로세스 또는 조직에 중요한 정보를 캡처합니다.

**이유가 여기에 있습니다**

사용자 지정 양식에 있는 불완전한 데이터는 작업을 지연시키고 보고에 영향을 줄 수 있습니다. 필수 필드 표시기(필드 이름 옆에 있는 빨간색 *)는 사용자에게 사용자 지정 양식을 편집하고 저장하거나 공식적으로 요청을 제출하기 전에 특정 정보가 필요하다는 것을 알려줍니다.

그러나 필수 필드는 드물게 신중하게 사용해야 합니다. 필요한 모든 필드가 있으면 사용자가 유용하고 완전한 정보로 필드를 채우지 못하게 할 수 있습니다. 또한 개체의 세부 정보 영역에서 사용자 지정 양식을 편집할 때 불완전한 필수 필드로 사용자 지정 양식이 저장되지 않습니다.

</br>
</br>

**우수 사례**

사용자 지정 양식에서 필드 이름을 변경할 때는 해당 필드를 호출하는 계산된 필드에 영향을 줄 수 있으므로 주의하십시오.

**이유가 여기에 있습니다**

필드 이름을 변경할 때 사용자 지정 양식의 계산된 사용자 지정 필드 또는 텍스트 모드로 기본 제공된 계산에서 사용할 위치를 업데이트해야 합니다. 필드 이름을 변경하면 계산이 중단되고 잘못된 정보가 발생합니다.

</br>
</br>

**우수 사례**

정기적으로 Workfront 시스템 유지 관리의 일부로 분기별로 한 번 등 사용자 지정 양식 및 필드를 정기적으로 검토합니다.

**이유가 여기에 있습니다**

사용자 지정 양식 및 필드는 조직에서 작업을 수행해야 하는 데이터를 수집하지 않는 경우에는 유용하지 않습니다.

업데이트를 만들 때 이러한 변경 사항이 Workfront의 다른 측면에 어떻게 영향을 주는지 염두에 두십시오. 예를 들어 필드 이름을 변경하면 해당 필드를 사용하는 계산이 손상될 수 있습니다. 또는 양식의 레이블이나 이름을 변경하면 보고서가 필요한 정보를 표시하지 않거나 다른 시스템과의 통합이 실행되지 않을 수 있습니다.

사용자 지정 양식을 업데이트하는 것 외에도, 사용이 낮거나 전혀 사용되지 않는 양식을 식별합니다. 자주 사용되지 않는 양식을 다른 양식과 결합할 수 있습니까? 또는 이제 해당 양식을 더 이상 사용하지 않을 차례입니다. 양식을 수집하는 정보가 더 이상 필요하지 않기 때문입니다.

이러한 불필요한 양식을 사용하여 작업을 수행하기 전에 사용자 지정 양식 및 사용자 지정 필드 삭제와 비활성화 방법에 대해 읽어 보십시오.

</br>
</br>

**우수 사례**

필요하지 않은 사용자 지정 양식을 삭제하는 대신 비활성화합니다.

**이유가 여기에 있습니다**

사용자 지정 양식을 삭제하면 해당 사용자 지정 양식을 통해 입력된 모든 사용자 지정 데이터가 삭제됩니다.
사용자 지정 양식을 비활성화하면 연결된 모든 이전 데이터가 유지됩니다. 즉, 이 정보에 대해 계속 보고할 수 있습니다.

또한 비활성화하면 사용자가 사용자 지정 양식을 선택하는 드롭다운 메뉴에 양식이 표시되지 않습니다. 그러나 양식은 이미 첨부된 모든 개체에 여전히 표시됩니다.

</br>
</br>

**우수 사례**

섹션 브레이크 아래의 사용자 지정 양식에서 더 이상 필요하지 않은 사용자 지정 필드를 숨깁니다. 그런 다음 시스템 관리자만 해당 섹션을 볼 수 있도록 합니다.

**이유가 여기에 있습니다**

기존 사용자 지정 필드를 삭제하면 사용자 지정 양식에서 제거될 뿐만 아니라, 사용자 지정 필드가 사용되는 모든 위치에서 해당 필드에 포함된 모든 데이터가 삭제됩니다. 즉, 보고서에 필요할 수 있는 이전 데이터가 삭제됩니다.

데이터 손실을 방지하려면 사용자 지정 양식 자체에서 불필요한 필드를 숨깁니다. 사용자 지정 양식의 섹션 나누기를 사용하면 사용자가 첨부된 Workfront 개체를 보거나, 기여하거나, 관리할 수 있는지 여부에 따라 해당 섹션의 일부인 필드를 숨기거나 표시할 수 있습니다. 또는 섹션을 관리자 전용으로 설정할 수 있으므로 시스템 관리자 액세스 수준을 가진 사용자만 양식의 전체 섹션을 볼 수 있습니다.

</br>
</br>

**우수 사례**

Workfront 인스턴스에서 사용자 지정 양식을 만들 수 있는 사용자를 제한합니다.


**이유가 여기에 있습니다**

사용자 지정 양식을 만들 수 있는 사용자 선택 그룹이 있으면 Workfront 인스턴스에서 만든 사용자 지정 양식의 수를 제어하는 데 도움이 됩니다.

또한 다른 사람이 양식을 작성할 수 있도록 허용하면 시스템 관리자의 작업이 완화되고 각 그룹이 정기적으로 사용하는 사용자 지정 양식 업데이트를 제어할 수 있습니다.
