# 모범 사례 - 요청 대기열

## Adobe Workfront “모범 사례”란 무엇입니까?

모범 사례는 효과적이고 효율적인 작업 과정을 나타내는 지침입니다. 사용자와 귀사의 사용자가 쉽게 채택할 수 있으며 조직 전체에서 복제할 수 있습니다.

이러한 권장 사항을 검토할 때 일부 Workfront 모범 사례는 보편적인 반면 다른 모범 사례는 주제에 더 구체적일 수 있음을 명심하십시오. Workfront 시스템을 올바르게 설정하고 사용하는 데 있어 이들 모범 사례를 프레임워크로 사용하십시오.

## 이 페이지 탐색

이 페이지를 스크롤하면 먼저 해당 주제에 대한 모든 모범 사례의 상위 수준 목록을 찾을 수 있습니다. 이를 통해 “이유”에 대한 세부 사항을 자세히 살펴보지 않고도 권장 사항을 검토할 수 있습니다.

“이러한 모범 사례를 채택하는 이유는 무엇입니까?” 영역(상위 수준의 목록 뒤)에서는 일부 모범 사례와 프로세스 및 도구 등으로 간주되는 이유에 대한 자세한 설명이 나와 있으며, Workfront 인스턴스를 사용하여 구현하는 것을 고려해야 합니다.

</br>
</br>

## 요청 대기열 모범 사례

* 요청 대기열의 각 요소(요청 대기열 프로젝트, 주제 그룹, 대기열 주제 및 라우팅 규칙)에 대한 설명이 나와 있습니다.

* 요청 대기열 프로젝트를 다른 프로젝트와 구별하기 위해 “현재”와 동일한 “요청 대기열” 또는 “운영”이라는 프로젝트 상태를 만듭니다.

* 대기열을 통해 제출된 요청과 함께 문제 승인을 사용하려는 경우, 거부됨이라는 문제 상태를 만듭니다.

* 요청 대기열에 “범용” 사용자 정의 양식을 할당하여 엔터프라이즈 수준의 일관된 데이터를 최대한 많이 캡처합니다.

* “모든 사람”과 요청 대기열을 공유하지 마십시오. 사용자가 필요에 맞는 대기열만 볼 수 있도록 대기열 세부 정보 설정을 설정합니다.

* 트래픽 관리자, 시스템 관리자 또는 할당된 사용자가 문제에 대해 직접 작업할 수 있도록 요청 대기열 보고서가 포함된 대시보드를 빌드하고 할당합니다.

* 대기열을 생성할 필요가 없는 사용자를 위해 레이아웃 템플릿을 사용하여 프로젝트의 왼쪽 패널 메뉴에서 요청 대기열 설정 옵션을 제거합니다.

* 사용자가 Workfront 관련 질문을 하고, 시스템 설정에 관련된 요청을 하고, 새 사용자 교육 일정을 잡을 수 있도록 시스템 관리자 요청 대기열을 만듭니다.

* 사용되지 않는 대기열을 확인하고 공유 해제하기 위해 정기적으로 요청 대기열을 감사합니다.

* 주제 그룹을 통해 요청 대기열에서 10개가 넘는 대기열 주제를 구성하여 더 짧고 관리하기 쉬운 목록을 만듭니다.

* 여러 대기열을 생성하는 대신 주제 그룹 및 대기열 주제로 요청 대기열을 분류하여 사용자가 사용할 수 있는 총 요청 대기열 수를 제어합니다.

* 각 대기열 주제에 대한 라우팅 규칙을 설정합니다. 최소한 기본 라우팅 규칙을 설정합니다.

* 선택적 라우팅이 필요한 경우, 주제 그룹 및 대기열 주제를 활용합니다.

* 개인이 아닌 팀에 요청을 라우팅합니다.

## 이러한 모범 사례를 채택하는 이유는 무엇입니까?


**모범 사례**

요청 대기열의 각 요소(요청 대기열 프로젝트, 주제 그룹, 대기열 주제 및 라우팅 규칙)에 대한 설명이 나와 있습니다.

**이유**

그룹 관리자, 향후 시스템 관리자 또는 요청 대기열을 유지 관리하는 기타 사용자는 설명을 통해 요청 대기열의 각 부분이 수행하는 작업을 정확히 알 수 있습니다.

설명 정보는 새 요청 창에서 필드의 정보 아이콘 위로 마우스를 가져가도 표시됩니다.

설명은 길 필요가 없으며, 요소의 목적이나 용도에 대한 간단한 주석만 있으면 됩니다.

</br>
</br>

**모범 사례**

요청 대기열 프로젝트를 다른 프로젝트와 구별하기 위해 “현재”와 동일한 “요청 대기열” 또는 “운영”이라는 프로젝트 상태를 만듭니다.

**이유**

요청 대기열은 프로젝트에 “활성화”되어 있고, 대기열이 활성화되려면 현재와 동일한 상태여야 합니다.

“현재” 상태의 실제 작업 프로젝트와 요청을 구분하기 위해 “요청 대기열” 또는 “운영”이라는 요청 대기열에서만 사용할 상태를 만듭니다. 그런 다음 이 상태를 사용하여 보고서를 작성하는 동안 요청 대기열 프로젝트를 제외하거나 포함할 수 있습니다.

</br>
</br>

**모범 사례**

문제 승인을 사용할 때 “거부됨”이라는 문제 상태를 만들고 거부된 경우 옵션을 “거부됨” 상태로 설정합니다.

**이유**

“거부됨” 상태를 사용하면 요청이 검토되고 거부되었음을 알 수 있습니다.

</br>
</br>

**모범 사례**

요청 대기열에 “범용” 사용자 정의 양식을 할당하여 엔터프라이즈 수준의 일관된 데이터를 최대한 많이 캡처합니다.

**이유**

“범용” 사용자 정의 양식은 제출되는 요청 유형에 관계없이 요청에 필요한 표준 정보를 수집합니다.

“범용” 사용자 정의 양식을 사용하면 만들고 유지 관리해야 하는 사용자 정의 양식의 수를 줄일 수 있습니다. 또한 모든 요청이 같은 방식으로 동일한 정보를 수집하여, 보고 및 데이터 분석의 일관성을 유지합니다.

</br>
</br>

**모범 사례**

“모든 사람”과 요청 대기열을 공유하지 마십시오.  사용자가 필요에 맞는 대기열만 볼 수 있도록 대기열 세부 정보 설정을 설정합니다.

**이유**

대부분의 경우 요청 대기열은 팀, 공급업체, 고객 등과 같은 특정 사람들과만 공유하면 됩니다. 요청자가 요청 대기열 목록에서 필요한 항목만 볼 수 있으면 항목을 쉽게 찾고 탐색할 수 있습니다.

</br>
</br>


**모범 사례**

트래픽 관리자, 시스템 관리자 또는 할당된 사용자가 문제에 대해 직접 작업할 수 있도록 요청 대기열 보고서가 포함된 대시보드를 빌드하고 할당합니다.

**이유**

사용자에게 들어오는 요청에 대한 빠르고 쉬운 액세스를 제공하면 셔플에서 작업이 손실되지 않습니다.

</br>
</br>

**모범 사례**

대기열을 생성할 필요가 없는 사용자를 위해 레이아웃 템플릿을 사용하여 프로젝트의 왼쪽 패널 메뉴에서 요청 대기열 설정 옵션을 제거합니다.


**이유**

이렇게 하면 모든 요청 대기열이 생성을 위한 적절한 프로세스(예: 거버넌스 위원회의 검토)를 거쳐 시스템 또는 그룹 관리자가 올바르게 설정할 수 있습니다.

또한 조직에 필요한 요청 유형에 집중하고 대기열 목록을 정리할 수 있습니다.

</br>
</br>

**모범 사례**

사용자가 Workfront 관련 질문을 하고, 시스템 설정에 관련된 요청을 하고, 새 사용자 교육 일정을 잡을 수 있도록 시스템 관리자 요청 대기열을 만듭니다.

**이유**

사용자가 질문을 제출하고 관리자가 Workfront 관련된 문제를 수집, 모니터링 및 응답할 수 있는 중앙 위치를 제공합니다.

또한 이 정보를 사용하여 시스템 관리자 역할의 시간, 노력 및 가치를 리더십에 보여 주고 추가 시스템 관리자에 대한 정당성을 입증합니다.

</br>
</br>


**모범 사례**

사용되지 않는 대기열을 확인하고 공유 해제하기 위해 정기적으로 요청 대기열을 감사합니다.

**이유**

Adobe Workfront 시스템의 설정 및 항목을 정기적으로 감사하면 시스템을 정리하고 불필요한 항목을 제거할 수 있습니다. 대기열이 더 이상 사용되거나 모니터링되지 않는 경우, 작업 요청이 무효화되지 않도록 사용자가 더 이상 대기열에 액세스할 수 없도록 합니다.

</br>
</br>


**모범 사례**

주제 그룹을 통해 요청 대기열에서 10개가 넘는 대기열 주제를 구성하여 더 짧고 관리하기 쉬운 목록을 만듭니다.

**이유**

주제 그룹은 선택할 초기 옵션 목록을 줄여서 사용자 채택을 늘리고 혼란을 줄입니다. 이를 통해 사용자가 요청을 제출하려고 할 때 당황하지 않고 원하는 것을 쉽게 찾을 수 있습니다.

또한 시스템 관리자 및/또는 요청 대기열 관리자는 사용자를 위한 원활한 탐색 경로를 생성하고, 제출되는 요청 유형을 더 효율적으로 구성하고 보고할 수 있습니다.

</br>
</br>

**모범 사례**

여러 대기열을 생성하는 대신 주제 그룹 및 대기열 주제로 요청 대기열을 분류하여 사용자가 사용할 수 있는 총 요청 대기열 수를 제어합니다.

**이유**

요청 대기열이 너무 많으면 사용자가 필요한 것을 찾기 어렵습니다.

또한 대기열이 적으면 트래픽 조정자, 시스템 관리자 또는 대기열을 관리하는 다른 사람들이 여러 요청 대기열 프로젝트를 탐색하지 않고도 필요한 정보를 더 빨리 찾을 수 있습니다.

요청 대기열마다 다른 액세스가 필요하거나 대기열 통합이 사용자에게 혼란을 줄 수 있는 경우, 여러 요청 대기열을 만듭니다.

</br>
</br>

**모범 사례**

각 대기열 주제에 대한 라우팅 규칙을 설정합니다. 최소한 기본 라우팅 규칙을 설정합니다.

**이유**

라우팅 규칙을 사용하면 수신 요청이 항상 할당되므로 작업이 중단되지 않습니다.

</br>
</br>

**모범 사례**

선택적 라우팅이 필요한 경우, 주제 그룹 및 대기열 주제를 활용합니다.

**이유**

라우팅 규칙은 사용자 정의 양식의 필드에 적용할 수 없습니다. 따라서 서로 다른 요청 유형을 서로 다른 팀/개인에게 라우팅해야 하는 경우, 작업이 적절하게 라우팅될 수 있도록 각 요청 유형을 고유한 주제 그룹/대기열 주제로 만듭니다.

</br>
</br>

**모범 사례**

개인이 아닌 팀에 요청을 라우팅합니다.

**이유**

팀에 요청이 전송되면 요청이 이루어지고 향후 작업에 수반될 수 있는 작업에 대한 가시성을 전체 팀에 제공합니다. 누구나 팀 페이지에서 새 항목을 확인하거나 대시보드의 보고서를 통해 새 항목을 추적할 수 있습니다.

또한 수신 요청을 검토하거나 할당하는 트래픽 관리자 또는 기타 사용자가 없는 경우, 백업을 자동으로 사용할 수 있으며 요청 정보에 액세스할 수 있습니다.
