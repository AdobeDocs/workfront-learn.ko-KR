# 우수 사례 - 요청 큐

## Adobe Workfront &quot;모범 사례&quot;란 무엇입니까?

모범 사례는 효과적이고 효율적인 작업 과정을 나타내는 지침입니다. 는 사용자와 회사의 사용자가 쉽게 채택할 수 있습니다. 또한 조직 전반에 걸쳐 성공적으로 복제할 수 있습니다.

이러한 권장 사항을 검토할 때 일부 Workfront 우수 사례는 일반적이지만, 다른 권장 사항은 주제에 더 구체적일 수 있음을 유의하십시오. 이러한 우수 사례를 프레임워크로 사용하여 Workfront 시스템 설정 및 사용을 안내할 수 있습니다.

## 이 페이지 탐색

이 페이지를 스크롤하면 먼저 주제에 대한 모든 우수 사례 목록이 표시됩니다. 이렇게 하면 &quot;이유&quot;의 세부 사항으로 이동하지 않고 권장 사항을 검토할 수 있습니다.

&quot;이러한 우수 사례는 무엇입니까?&quot;가 상위 수준 목록 다음에 있는 영역은 일부 우수 사례에 더 자세한 정보를 제공하고 프로세스, 도구 등으로 간주되는 이유를 Workfront 인스턴스로 구현하는 것이 좋습니다.

## 요청 큐 우수 사례

* 요청 큐 프로젝트, 항목 그룹, 큐 항목 및 라우팅 규칙 등 요청 큐의 각 요소에 대한 설명을 포함합니다.

* 요청 큐 프로젝트를 다른 프로젝트와 구분하기 위해 &quot;현재&quot;와 일치하는 &quot;요청 큐&quot; 또는 &quot;작동&quot;이라는 프로젝트 상태를 만듭니다.

* 큐를 통해 제출된 요청에 문제 승인을 사용할 계획이라면 거부됨이라는 문제 상태를 만듭니다.

* &quot;범용&quot; 사용자 지정 양식을 요청 큐에 할당하여 가능한 한 많은 엔터프라이즈 수준의 일관된 데이터를 캡처합니다.

* &quot;모든 사용자&quot;와 요청 큐를 공유하지 마십시오. 사용자가 자신의 요구 사항과 관련된 대기열만 표시되도록 큐 세부 정보 설정을 설정하십시오.

* 트래픽 관리자, 시스템 관리자 또는 할당된 사용자가 문제에 직접 작업할 수 있도록 요청 큐 보고서가 포함된 대시보드를 작성하고 할당합니다.

* 레이아웃 템플릿을 사용하여 큐를 만들 필요가 없는 사용자를 위해 프로젝트의 왼쪽 패널 메뉴에서 요청 큐 설정 옵션을 제거합니다.

* 사용자가 Workfront 관련 질문을 하고, 시스템 설정에 대한 요청을 수행하고, 새 사용자 교육 일정을 잡을 수 있도록 시스템 관리자 요청 큐를 만듭니다.

* 사용되지 않는 대기열을 식별하고 공유 취소할 수 있도록 정기적으로 감사 요청 큐를 제공합니다.

* 항목 그룹을 사용하여 요청 큐에서 10개 이상의 큐 항목을 구성하여 목록을 보다 빠르고 쉽게 만들 수 있습니다.

* 여러 개의 큐를 만드는 대신 주제 그룹과 큐 항목을 사용하여 요청 큐를 분류하여 사용자가 사용할 수 있는 총 요청 큐 수를 제어합니다.

* 각 대기열 항목에 대한 라우팅 규칙을 설정합니다. 최소한 기본 공정순서 규칙을 설정합니다.

* 선택적 라우팅이 필요할 때 주제 그룹 및 대기열 항목을 활용하십시오.

* 개인이 아닌 팀에 요청을 라우팅합니다.

## 이러한 우수 사례는 무엇입니까?


**우수 사례**

요청 큐 프로젝트, 항목 그룹, 큐 항목 및 라우팅 규칙 등 요청 큐의 각 요소에 대한 설명을 포함합니다.

**이유가 여기에 있습니다**

설명을 사용하면 그룹 관리자, 향후 시스템 관리자 또는 요청 큐를 유지하는 다른 사용자가 요청 큐의 각 작업을 정확하게 파악할 수 있습니다.

설명 정보는 새 요청 창의 필드에 있는 정보 아이콘을 마우스로 가리키면 나타납니다.

설명은 오래 사용할 필요가 없으며 요소의 용도와 사용에 대한 간단한 언급일 뿐입니다.

</br>
</br>

**우수 사례**

요청 큐 프로젝트를 다른 프로젝트와 구분하기 위해 &quot;현재&quot;와 일치하는 &quot;요청 큐&quot; 또는 &quot;작동&quot;이라는 프로젝트 상태를 만듭니다.

**이유가 여기에 있습니다**

큐가 활성화되려면 요청 큐 &quot;라이브&quot; 상태가 현재 상태여야 합니다.

요청을 &quot;현재&quot; 상태의 실제 작업 프로젝트와 구분하려면 &quot;요청 큐&quot; 또는 &quot;작동&quot; 이라는 요청 대기열에서만 사용할 상태를 만드십시오. 그런 다음 이 상태를 사용하여 보고서를 작성하는 동안 요청 큐 프로젝트를 제외하거나 포함할 수 있습니다.

</br>
</br>

**우수 사례**

문제 승인을 사용할 때 &quot;거부됨&quot;이라는 문제 상태를 만들고 &quot;거부된 경우&quot; 옵션을 &quot;거부됨&quot; 상태로 설정합니다.

**이유가 여기에 있습니다**

&quot;거부됨&quot; 상태를 사용하면 요청이 검토 및 거부되었음을 알 수 있습니다.

</br>
</br>

**우수 사례**

&quot;범용&quot; 사용자 지정 양식을 요청 큐에 할당하여 가능한 한 많은 엔터프라이즈 수준의 일관된 데이터를 캡처합니다.

**이유가 여기에 있습니다**

범용 사용자 지정 양식은 제출하는 요청 유형에 관계없이 요청에 필요한 표준 정보를 수집합니다.

범용 사용자 지정 양식이 있으면 만들고 유지 관리하는 데 필요한 사용자 지정 양식 수를 줄이는 데 도움이 됩니다. 또한 모든 요청이 동일한 방식으로 동일한 정보를 수집하여 일관된 보고 및 데이터 분석을 가능하게 합니다.

</br>
</br>

**우수 사례**

&quot;모든 사용자&quot;와 요청 큐를 공유하지 마십시오.  사용자가 자신의 요구 사항과 관련된 대기열만 표시되도록 큐 세부 정보 설정을 설정하십시오.

**이유가 여기에 있습니다**

대부분의 경우 요청 큐는 팀, 공급업체, 고객 등과 같은 특정 사용자 집합과 공유만 하면 됩니다. 요청자가 요청 큐 목록에서 필요한 것만 확인하면 쉽게 찾고 탐색할 수 있습니다.

</br>
</br>


**우수 사례**

트래픽 관리자, 시스템 관리자 또는 할당된 사용자가 문제에 직접 작업할 수 있도록 요청 큐 보고서가 포함된 대시보드를 작성하고 할당합니다.

**이유가 여기에 있습니다**

사용자에게 들어오는 요청에 빠르고 쉽게 액세스할 수 있도록 허용하면 작업이 셔플에서 손실되지 않습니다.

</br>
</br>

**우수 사례**

레이아웃 템플릿을 사용하여 큐를 만들 필요가 없는 사용자를 위해 프로젝트의 왼쪽 패널 메뉴에서 요청 큐 설정 옵션을 제거합니다.


**이유가 여기에 있습니다**

이렇게 하면 모든 요청 큐가 적절한 작성 프로세스(예: 거버넌스 위원회의 검토)를 거치고 시스템 또는 그룹 관리자가 올바로 설정되도록 할 수 있습니다.

또한 큐 목록을 구성하고 조직에서 필요로 하는 요청 유형에 중점을 두는 데 도움이 됩니다.

</br>
</br>

**우수 사례**

사용자가 Workfront 관련 질문을 하고, 시스템 설정에 대한 요청을 수행하고, 새 사용자 교육 일정을 잡을 수 있도록 시스템 관리자 요청 큐를 만듭니다.

**이유가 여기에 있습니다**

사용자가 질문을 제출하고 관리자가 Workfront 관련 문제를 수집, 모니터링 및 응답할 수 있는 중앙 집중식 위치를 제공합니다.

또한 이 정보를 사용하여 시스템 관리자 역할의 시간, 노력 및 가치를 지도력과 추가 시스템 관리자를 위한 근거로서 표시할 수 있습니다.

</br>
</br>


**우수 사례**

사용되지 않는 대기열을 식별하고 공유 취소할 수 있도록 정기적으로 감사 요청 큐를 제공합니다.

**이유가 여기에 있습니다**

Adobe Workfront 시스템의 설정 및 항목에 대한 정기적인 감사를 통해 불필요한 항목을 혼란스럽고 자유롭게 유지할 수 있습니다. 큐가 더 이상 사용되지 않거나 모니터링되지 않는 경우 사용자가 더 이상 큐에 액세스할 수 없으므로 작업 요청이 무효가 되지 않도록 하십시오.

</br>
</br>


**우수 사례**

항목 그룹을 사용하여 요청 큐에서 10개 이상의 큐 항목을 구성하여 목록을 보다 빠르고 쉽게 만들 수 있습니다.

**이유가 여기에 있습니다**

주제 그룹은 사용자 채택을 늘리고 선택할 초기 옵션 목록을 줄여 혼동을 줄입니다. 이를 통해 사용자는 요청을 제출하려고 할 때 압도되지 않고도 원하는 항목을 손쉽게 찾을 수 있습니다.

또한 시스템 관리자 및/또는 요청 대기열 관리자에게 사용자가 원활하게 탐색할 수 있는 경로 및 더 나은 방법을 만들어 제출할 요청 유형을 구성하고 보고할 수 있는 기회를 제공할 수 있습니다.

</br>
</br>

**우수 사례**

여러 개의 큐를 만드는 대신 주제 그룹과 큐 항목을 사용하여 요청 큐를 분류하여 사용자가 사용할 수 있는 총 요청 큐 수를 제어합니다.

**이유가 여기에 있습니다**

요청 큐가 너무 많아 사용자가 필요한 항목을 찾기가 어렵습니다.

큐 수를 줄이면 트래픽 조정자, 시스템 관리자 또는 대기열을 관리하는 다른 사람이 작업을 보다 신속하게 찾을 수 있으므로 여러 요청 큐 프로젝트로 이동할 필요 없이 보다 신속하게 정보를 찾을 수 있습니다.

다른 요청 큐에 대해 다른 액세스가 필요하거나 통합 큐가 사용자에게 혼동을 줄 경우 여러 요청 큐를 만듭니다.

</br>
</br>

**우수 사례**

각 대기열 항목에 대한 라우팅 규칙을 설정합니다. 최소한 기본 공정순서 규칙을 설정합니다.

**이유가 여기에 있습니다**

라우팅 규칙은 작업이 균열을 받지 않도록 사람이 항상 수신 요청을 지정하도록 합니다.

</br>
</br>

**우수 사례**

선택적 라우팅이 필요할 때 주제 그룹 및 대기열 항목을 활용하십시오.

**이유가 여기에 있습니다**

라우팅 규칙은 사용자 지정 양식의 필드에 적용할 수 없습니다. 따라서 서로 다른 유형의 요청을 서로 다른 팀/개인에게 라우팅해야 하는 경우, 작업을 제대로 라우팅할 수 있도록 각 유형의 요청을 고유한 주제 그룹/큐 항목으로 만듭니다.

</br>
</br>

**우수 사례**

개인이 아닌 팀에 요청을 라우팅합니다.

**이유가 여기에 있습니다**

요청이 팀에 보내지면, 그것은 모든 팀에게 요청과 다가오는 일이 무엇을 수반할 것인지에 대한 가시성을 제공한다. 모든 사람이 팀 페이지에서 새 항목을 보거나 대시보드의 보고서를 통해 새로운 기능을 추적할 수 있습니다.

또한, 트래픽 관리자 또는 들어오는 요청을 검토 또는 지정하는 다른 담당자를 사용할 수 없을 때 자동으로 백업을 사용할 수 있고 요청 정보에 액세스할 수 있도록 합니다.
