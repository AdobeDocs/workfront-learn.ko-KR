# 모범 사례 - 프로젝트 템플릿

## Adobe Workfront &quot;모범 사례&quot;란 무엇입니까?

모범 사례는 효과적이고 효율적인 작업 과정을 나타내는 지침입니다. 사용자와 회사의 사용자가 쉽게 채택할 수 있으며 조직 전반에서 성공적으로 복제할 수 있습니다.

이러한 권장 사항을 검토할 때 일부 Workfront 우수 사례는 보편적이지만 다른 우수 사례는 주제에 더 구체적일 수 있다는 점을 유의하십시오. 이러한 모범 사례를 프레임워크로 사용하여 Workfront 시스템 설정 및 사용을 안내합니다.

## 이 페이지 탐색

이 페이지를 스크롤하면 먼저 주제에 대한 모든 모범 사례에 대한 높은 수준의 목록이 표시됩니다. 이렇게 하면 &quot;이유&quot;에 대한 세부 정보로 이동하지 않고 권장 사항을 검토할 수 있습니다.

&quot;이러한 모범 사례는 왜 있습니까?&quot; 상위 수준 목록 다음에 나오는 영역에서는 몇 가지 모범 사례와 이러한 모범 사례를 프로세스, 도구 등으로 간주하는 이유에 대해 자세히 설명합니다. Workfront 인스턴스를 사용하여 구현을 고려해야 합니다.

</br>
</br>

## 프로젝트 템플릿 모범 사례

* 프로젝트를 만들 때 템플릿을 사용하십시오.

* 프로젝트 템플릿에 대한 이름 지정 규칙을 설정합니다.

* 프로젝트 템플릿을 만들고 업데이트할 수 있는 사용자 선택 그룹을 설정합니다.

* 프로젝트 템플릿에서 프로젝트 공유 를 사용하여 해당 템플릿을 사용하여 만든 프로젝트에 대한 액세스 권한을 자동으로 부여합니다.

* 작업 역할 또는 팀을 개인이 아닌 작업에 할당합니다.

* 프로젝트 템플릿에서 작업을 생성할 때 너무 세부적으로 증가하지 마십시오. 프로젝트 템플릿의 작업 수를 작업을 완료하는 데 필요한 작업으로 제한합니다.

* 여러 작업으로 분류하지 않고 작업 설명을 사용하여 작업의 작은 단계를 캡처합니다.

* 템플릿 작업에 작업 기간, 계획된 시간 및 전임 작업이 포함되어 있는지 확인합니다.

* 프로젝트 세부 정보를 미리 구성하고 템플릿에 사용자 정의 양식을 첨부합니다.

* 정기적으로 프로젝트 템플릿을 검토하고 업데이트합니다.

* 템플릿을 공유하거나 다른 사용자가 사용하기 전에 필요한 모든 정보가 있는지 템플릿을 확인하십시오.

* 템플리트에서 일정 시작 옵션을 조정할 때 태스크 제한조건을 검토하고 갱신합니다.

* 템플릿에서 프로젝트 팀을 확인하고 프로젝트와 연결되지 않는 사용자를 제거합니다.

## 이러한 모범 사례는 무엇입니까?

**모범 사례**

프로젝트를 만들 때 템플릿을 사용하십시오.

**그 이유는 다음과 같습니다**

프로젝트 템플릿은 프로젝트에 포함해야 하는 작업, 타임라인 구성 방법 등에 대한 프로젝트 관리자(및 프로젝트를 만드는 다른 사람)의 추측을 제거합니다. 템플릿은 프로젝트 생성 속도를 높이는 가장 효과적인 방법입니다.

중요한 것은 템플릿은 유사한 유형의 프로젝트에서 일관성을 유지하기 때문에 인력, 프로세스 및 데이터 포인트가 항상 동일한 방법으로 상세화된다는 것입니다. 빠른 전환(1일 또는 2일)과 최소한의 작업이 있는 프로젝트라도 프로젝트 템플릿을 사용하여 만들면 도움이 될 수 있습니다.

프로젝트 간의 이러한 일관성은 팀, 부서 및 조직에 대한 의사 결정을 내릴 때 중요한 더 정확한 데이터를 생성합니다.

</br>
</br>

**모범 사례**

프로젝트 템플릿에 대한 이름 지정 규칙을 설정합니다.

**그 이유는 다음과 같습니다**

일관된 이름 지정을 사용하면 템플릿을 쉽게 찾을 수 있습니다. 또한 여러 팀 또는 부서에 이름이 비슷한 템플릿이 있는 경우 프로젝트를 만드는 프로젝트 관리자 및 다른 사람이 올바른 템플릿을 선택하는 데 도움이 됩니다.

</br>
</br>

**모범 사례**

프로젝트 템플릿을 만들고 업데이트할 수 있는 사용자 선택 그룹을 설정합니다.

**그 이유는 다음과 같습니다**

일관되고 잘 구축된 프로젝트 템플릿을 통해 작업을 효과적으로 관리하고 정확한 보고를 수행할 수 있습니다. 실수로 또는 승인되지 않은 변경을 방지하기 위해 템플릿을 편집할 수 있는 사용자 수를 제한하십시오.

</br>
</br>

**모범 사례**

프로젝트 템플릿에서 프로젝트 공유 를 사용하여 해당 템플릿을 사용하여 만든 프로젝트에 대한 액세스 권한을 자동으로 부여합니다.

**그 이유는 다음과 같습니다**

특정 프로젝트에 대한 액세스 권한은 프로젝트 자체를 통해 부여됩니다. 동일한 사용자 그룹이 항상 특정 템플릿으로 만든 프로젝트에 액세스해야 하는 경우 템플릿의 프로젝트 공유 옵션 아래에 추가하십시오. 프로젝트가 생성되면 바로 프로젝트에 대한 액세스를 제어할 수 있을 뿐만 아니라 향후 권한을 변경해야 하는 경우 확장성 노력을 간소화합니다.

**참고**: 템플릿 공유는 템플릿 자체에 대한 액세스 권한을 부여합니다. 템플릿으로 프로젝트를 만들려면 사용자에게 보기 권한 이상이 있어야 합니다.

</br>
</br>

**모범 사례**

작업 역할 또는 팀을 개인이 아닌 작업에 할당합니다.

**그 이유는 다음과 같습니다**

개별 사용자가 Position을 변경하거나 조직을 떠나면 해당 사용자를 포함하는 프로젝트 템플릿을 수동으로 업데이트해야 합니다. 시스템 또는 그룹 관리자 또는 프로젝트 관리자의 일부에서 시간이 소요됩니다.

템플릿에서 작업 역할이나 팀을 사용하는 경우, 해당 작업 역할이나 팀에 할당된 사용자가 작업을 할당할 수 있기 때문에 직원 교체 작업은 프로젝트 템플릿에 직접적인 영향을 미치지 않습니다. 이렇게 하면 균열을 통해 작업이 미끄러지지 않도록 할 수 있습니다. 또한 Workfront은 작업 역할에 할당된 사람 목록을 보여 줄 수 있으므로 작업 역할 할당을 통해 개별 사용자에게 작업을 보다 쉽게 할당할 수 있습니다.

또한 작업 역할은 Workfront의 리소스 계획 도구에서 필요한 리소스를 계산하고 향후 작업을 계획하는 데 도움이 됩니다.

</br>
</br>

**모범 사례**

프로젝트 템플릿에서 작업을 생성할 때 너무 세부적으로 증가하지 마십시오. 프로젝트 템플릿의 작업 수를 작업을 완료하는 데 필요한 작업으로 제한합니다.

**그 이유는 다음과 같습니다**

프로젝트 템플릿이 너무 복잡하면 프로젝트 관리자, 리소스 관리자, 팀원 등 사용자에게 좋지 않은 경험이 발생합니다. 작업이 너무 많으면 작업 기한이 겹치고 동일한 작업 역할이나 개인에게 여러 작업이 할당되어 프로젝트 타임라인을 관리하기가 어렵습니다.


</br>
</br>

**모범 사례**

여러 작업으로 분류하지 않고 작업 설명을 사용하여 작업의 작은 단계를 캡처합니다.

**그 이유는 다음과 같습니다**

한 행에 있는 여러 작업이 동일한 작업 역할/개인에게 지정되는 경우, 이는 해당 작업이 결합될 수 있음을 나타냅니다. 사용자에게 너무 많은 작업이 할당되면 완료할 작업이 더 많은 것처럼 느껴질 수 있으며, 이는 Workfront 채택에 영향을 줄 수 있습니다.

</br>
</br>

**모범 사례**

템플릿 작업에 작업 기간, 계획된 시간 및 전임 작업이 포함되어 있는지 확인합니다.

**그 이유는 다음과 같습니다**

기간, 계획된 시간 및 전임 작업 등 세 가지 사항은 프로젝트 타임라인의 기본 구성단위입니다. 이것은 작업이 얼마나 오래 걸리고 언제 수행되어야 하는지를 아는 데 핵심입니다. 리소스 용량, 가용성 등을 계산하기 위해 Workfront의 리소스 관리 도구 지속 시간, 계획된 시간 및 작업 역할 할당을 더합니다.

기간 또는 계획된 시간을 처음 예상하는 방법을 잘 모를 경우 프로젝트 팀과 협력하여 초기 예상 값을 정의하십시오. 템플릿을 사용한 후에는 프로젝트 팀과 다시 만나 템플릿을 더 정확하게 만들기 위해 변경할 수 있는 위치를 결정합니다. 사용자가 Workfront에서 시간을 기록하는 경우 프로젝트의 계획된 시간과 실제 시간을 비교하여 조정이 필요한 위치를 확인할 수 있습니다.


</br>
</br>

**모범 사례**

프로젝트 세부 정보를 미리 구성하고 템플릿에 사용자 정의 양식을 첨부합니다.

**그 이유는 다음과 같습니다**

모든 프로젝트에 표준인 정보를 프로젝트 템플릿에 입력해야 합니다. 이렇게 하면 프로젝트 생성 속도를 높이는 데 도움이 될 뿐만 아니라 필요한 정보가 존재하고 프로젝트 간에 일관성을 유지할 수 있습니다.

템플릿을 사용하여 요청을 프로젝트로 전환할 때 제출된 정보를 가져오려면 요청 사용자 정의 양식과 일치하는 프로젝트 사용자 정의 양식을 첨부하십시오.

</br>
</br>

**모범 사례**

정기적으로 프로젝트 템플릿을 검토하고 업데이트합니다.

**그 이유는 다음과 같습니다**

프로세스 및 팀이 변경되면 프로젝트 템플릿을 업데이트해야 합니다. 분기별 등 정기적인 케이던스를 설정하여 활발하게 사용되지 않는 템플릿을 확인합니다. 비활성화할 수 있으므로 아직 Workfront에 있지만 템플릿 선택 목록에 표시되지 않습니다.

</br>
</br>

**모범 사례**

템플릿을 공유하거나 다른 사용자가 사용하기 전에 필요한 모든 정보가 있는지 템플릿을 확인하십시오.


**그 이유는 다음과 같습니다**

템플릿은 프로젝트를 만드는 데 반복해서 사용되므로 모든 것이 적절하고 완전하게 설정되었는지 확인해야 합니다. 이렇게 하면 모든 프로젝트에 일관성이 유지되며 보고를 위한 정확한 데이터가 제공됩니다.

기간 및 계획된 시간과 같은 작업 설정 외에도 템플릿을 공유하기 전에 검토할 사항은 다음과 같습니다.

* 일정 시작 설정
* 작업 제한
* 프로젝트 소유자, 스폰서, 그룹 및 회사
* Portfolio 및 프로그램
* 마일스톤 경로 및 단계
* 승인 프로세스
* 프로젝트에서 작업에 할당된 사용자에게 프로젝트에 대한 기여 액세스 권한이 있는지 확인
* 리소스 풀
* 미리 알림
* 예약
* 통화 설정(해당되는 경우)
* 표준 문서 첨부
* 필요한 세관 양식 첨부
* 추가 인력이 할당되지 않았는지 프로젝트 팀에 확인

</br>
</br>

**모범 사례**

템플리트에서 일정 시작 옵션을 조정할 때 태스크 제한조건을 검토하고 갱신합니다.

**그 이유는 다음과 같습니다**

프로젝트에 서로 다른 작업 제한을 혼합하면 예상치 못한 혼란스러운 계획 날짜 계산이 발생할 수 있습니다. 예를 들어 일정 시작 일자 옵션에 대해 시작 일자를 선택하면 해당 프로젝트에서 생성된 모든 작업은 기본적으로 가능한 한 빨리 작업 제한 사항에 할당됩니다. 나중에 일정 시작 옵션을 완료 일자로 전환하는 경우 생성된 모든 작업에는 기본적으로 가능한 한 늦게 작업 제한이 있습니다. 이러한 각 제한 사항에 의도하지 않게 작업이 혼합되면 프로젝트 타임라인에서 계획된 일자가 혼란스러울 수 있습니다.


</br>
</br>

**모범 사례**

템플릿에서 프로젝트 팀을 확인하고 프로젝트로 작업하지 않는 사용자를 제거합니다.

**그 이유는 다음과 같습니다**

기존 프로젝트에서 템플릿을 만들면 프로젝트에서 작업/문제가 할당된 직원을 가져옵니다. 템플릿을 사용하여 작업할 때 이전에 할당된 작업을 삭제하거나 템플릿에서 직접 수행한 할당을 변경할 수 있습니다.

이러한 모든 사용자는 프로젝트의 사람 및 예약 섹션에 프로젝트 팀의 일부로 나열됩니다. 그 결과 해당 템플릿에서 만든 모든 프로젝트에 전파됩니다. 이렇게 하면 프로젝트 팀의 일부로 프로젝트 활동에 대한 알림을 받고, 내가 진행 중인 프로젝트 목록에서 프로젝트를 확인하고, 프로젝트와 해당 작업, 문제 및 문서에 대한 권한을 얻을 수 있으므로 사용자에게 혼동이 발생할 수 있습니다.
